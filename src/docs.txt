🧪 QUALITY ANALYST (QA) BASICS
💡 What is software testing?

Answer:

Software testing is the process of verifying that an application works as expected and identifying 
bugs or issues before it is released to users.

🔍 Types of testing (they LOVE this)

Answer:

There are several types of testing:

Unit testing → testing individual functions/components
Integration testing → testing how different parts work together
System testing → testing the entire application
User Acceptance Testing (UAT) → ensuring it meets user requirements
🐞 What is a bug?

Answer:

A bug is an error or flaw in the software that causes it to 
produce incorrect or unexpected results.

✅ What is a test case?

Answer:

A test case is a set of conditions or steps used to verify that a 
specific feature works correctly.

🚀 What is regression testing?

Answer:

Regression testing ensures that new changes or features 
do not break existing functionality.

⚡ Manual vs Automated testing

Answer:

Manual testing is done by a human without tools, 
while automated testing uses scripts and tools to run tests automatically.
Automation is faster and useful for repeated tests.

⚙️ CI/CD + GITHUB ACTIONS
💡 What is CI/CD?

Answer:

CI/CD stands for Continuous Integration and Continuous Deployment.

##CI
Continuous Integration means automatically 
testing and merging code changes frequently.

### CD
Continuous Deployment means automatically 
releasing code to production after it passes tests.

🔧 What is GitHub Actions?

Answer:

GitHub Actions is a CI/CD tool that allows you to automate workflows like building, testing, and 
deploying code directly from a GitHub repository.

🔄 What is a workflow?

Answer:

A workflow is a defined automated process in GitHub Actions that runs when certain events happen, like pushing code or creating a pull request.

🧱 What are jobs and steps?

Answer:

A workflow contains jobs, and each job contains steps.
Steps are individual tasks like installing dependencies or running tests.

🚦 When does a workflow run?

Answer:

A workflow runs based on triggers such as:

push to a branch
pull request
manual trigger
❌ What happens if a build fails?

Answer:

If a build fails, the workflow stops, and the code should not be merged or 
deployed until the issue is fixed.

🔥 Common practical question
“How would you ensure code quality before deployment?”

Answer:

I would set up a CI pipeline that runs automated tests, 
and builds the application whenever code is pushed.
This ensures that only code that passes all checks is merged and deployed.

🧠 QUICK 30-SECOND SUMMARY (READ BEFORE INTERVIEW)
QA = finding bugs + ensuring quality
Test types = unit, integration, system, UAT
CI/CD = automate testing + deployment
GitHub Actions = tool for workflows
Workflow → jobs → steps
Always ensure tests pass before deploy



💡 What is End-to-End (E2E) testing?

Answer:

End-to-End testing is a type of testing where the entire application flow is tested from start to finish, just like a real user would use the system.

🔍 Simple example (THIS is what they want)

Example:

For example, in a login system, an E2E test would:

Open the application
Enter user credentials
Click login
Verify that the user is successfully logged in and redirected to the dashboard
💻 Fullstack example (you can use this 🔥)

In a blog application, an End-to-End test could involve:

A user signs up
Logs in
Creates a post
Views the post on the homepage

This ensures the frontend, backend, and database all work together correctly.

⚖️ Difference from other tests (quick flex)

Unlike unit tests that check individual functions, E2E tests verify the entire system flow and user experience.

🧠 One-liner to remember

E2E testing simulates real user behavior to ensure the whole system works together correctly.

🔥 Bonus (if they go deeper)
“What tools can you use for E2E testing?”
Cypress,Selenium,Playwright


🧪 Big Picture: Where E2E Testing Belongs

E2E is part of:

Software Testing → Testing Types → End-to-End Testing

📊 The testing “levels” (this is what interviewers expect)

Think of it like layers:

1️⃣ Unit Testing (foundation)
Tests small pieces of code
Example: a function that adds numbers
2️⃣ Integration Testing
Tests how parts work together
Example: API + database
3️⃣ System Testing
Tests the entire application as a whole
4️⃣ End-to-End (E2E) Testing 🚀
Tests the full user journey from start to finish
5️⃣ User Acceptance Testing (UAT)
Done by users to confirm it meets real needs
🧠 Simple way to explain in an interview

Answer:

End-to-End testing is one of the higher-level testing types in software testing. It comes after unit and integration testing and focuses on verifying the complete user flow across the entire system.

🔥 Visual way (say it like this if you want to sound sharp)

Testing starts from small units, then moves to integration, then the full system, and finally End-to-End testing, which simulates real user behavior.

⚡ Key insight (this is what makes you stand out)

E2E tests are fewer but very important because they validate real-world usage, while unit tests are many and focus on individual components.

🎯 One-liner to lock it in

E2E testing sits at the top of the testing levels and validates the complete application flow from a user’s perspective.
